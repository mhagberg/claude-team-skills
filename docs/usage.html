Install

One-time setup per machine. Symlinks the skills into ~/.claude/skills/ so every Claude Code session sees them.

git clone https://github.com/mhagberg/claude-team-skills.git ~/claude-team-skills
cd ~/claude-team-skills
./install.sh
The installer is safe: if a skill of the same name already exists in ~/.claude/skills/, it is moved aside to <name>.bak.<timestamp>/ before the new symlink is created. Nothing is silently overwritten.

Available skills

/parallel-code-review ready

5 independent agents review the current diff (or a GitHub PR) in parallel, each with a different lens. Consolidated table, rating spread, unanimous concerns, contested calls, recommendation.

Invocations

/parallel-code-review              # current branch diff vs main
/parallel-code-review 1234         # GitHub PR #1234 (uses gh pr diff)
/parallel-code-review src/foo.py   # scoped to one file or directory

The 5 lenses

#LensFocus
1CorrectnessBugs, off-by-ones, wrong conditions, broken invariants, incorrect math/SQL
2SecurityInjection, authz/authn, secret leakage, unsafe deserialization, path traversal
3Error handlingUnhandled exceptions, swallowed errors, missing null/empty/boundary, partial failures
4ReadabilityNaming, dead code, premature abstraction, comments that lie, oversized functions
5TestingMissing tests, weak assertions, tests that test the mock, untested branches

/parallel-plan-review ready

5 independent agents review one or more plan/spec files in parallel. Auto-discovers PLAN.md, plans/*.md, ai_docs/plans/**, docs/plans/**, and **/PROJECT_PROCESS.md when no path is given.

Invocations

/parallel-plan-review              # auto-discover all plan files
/parallel-plan-review docs/plans/  # scope to a directory
/parallel-plan-review PLAN.md      # scope to one file

The 5 lenses

#LensFocus
1Goal alignmentDoes the plan solve the stated problem? Scope drift, unstated assumptions
2Technical soundnessArchitecture, dependencies, sequencing — will this actually work?
3Risk & rollbackWhat happens when a step fails? Reversibility, blast radius, partial-completion states
4TestabilityHow will we know each step worked? Are success criteria measurable?
5Effort & decompositionSteps too coarse / too fine? Missing prerequisites? Hidden dependencies?

Final recommendation lands on one of: execute-as-written, execute-with-revisions, rework.

/git-push ready

Stage all, write a conventional-commit message from the diff, commit, push (creates upstream if needed).

Refuses to run on: main / master; dirty .env, *.pem, *.key, credentials.json, *secret* files. Never --force, --no-verify, or --amend.

/git-push-deploy ready

/git-push plus running the repo's .claude-deploy command after the push succeeds.

.claude-deploy file format

# One branch pattern per line, first match wins
main: bun run deploy:prod
preview/*: bun run deploy:preview
*: bun run deploy:branch

No .claude-deploy = the skill stops with a message before pushing. Use /git-push if you want push-only.

Hard-excluded paths (skill refuses to run, even if you have a .claude-deploy): anything under /odoocker/ or /odoo-*, repos with odoo-bin at root, and any */sage_dbt or */dbt_rollup. Those pipelines are owned by external automation.

/git-ship ready

/git-push-deploy plus opening a PR and auto-merging once CI is green. Falls back to --ff-only if no GitHub remote.

  1. Stage, conventional commit, push.
  2. gh pr create --fill --base <default-branch> (or reuse existing).
  3. gh pr merge --squash --auto --delete-branch — waits for required checks.
  4. After merge: checkout default branch, pull, run .claude-deploy for that branch.

Pass --no-deploy if the repo has no .claude-deploy and you just want to merge. Polls for merge state up to 15 minutes; if BLOCKED (failed CI, conflicts, required reviews), halts and reports — never bypasses branch protection.

/git-ship-all ready

Highest-blast-radius skill. Discovers every dirty repo (parent + submodules), runs a pre-flight 5-agent parallel review across the combined diffs, asks for one batched go/no-go, then ships each repo via the /git-ship flow. Parent submodule bumps land last.

Flags

  • --dry-run — discover and review only; no push, merge, or deploy.
  • --fire — skip the human go/no-go prompt. Unanimous BLOCKERs from the review still halt it.
Conflicts ALWAYS halt — no auto-resolution. If any repo's PR merge reports BLOCKED (conflicts, failed CI, required reviews), the whole run stops at that repo. Already-shipped repos stay shipped. No -X ours/theirs, no accept-incoming, no force push. The design premise is that silent auto-resolution destroys code.

Discovery example

| Repo                          | Branch     | Dirty files | Unpushed | Status   |
|-------------------------------|------------|------------:|---------:|----------|
| <parent>                      | feature/x  |          3  |       2  | dirty    |
| dataxcel-board-reports-...    | main       |          0  |       0  | clean    |
| odoo-bank-reconciliation      | feature/y  |          7  |       0  | dirty    |
| odoocker                      | -          |          -  |       -  | excluded |

How to use a skill

In any Claude Code session, type the slash command. Claude will load the skill's instructions for that turn, spawn the parallel reviewers, and print the report.

$ claude
> /parallel-code-review
Reviewing 7 files / +312 / -88 from current branch vs main.
[spawning 5 reviewers in parallel...]

Share with your team

The repo is public at github.com/mhagberg/claude-team-skills. Teammates run the same three install commands.

  1. Clone git clone https://github.com/mhagberg/claude-team-skills.git ~/claude-team-skills

Add a new skill

  1. Create skills/<your-skill>/SKILL.md with YAML frontmatter: 
    ---
name: your-skill
description: One-line summary shown in the skill picker.
---

Instructions to Claude in plain prose. The body must be self-contained — Claude
will not have access to other files in this repo when the skill runs.
  2. Run ./install.sh again — it's idempotent and picks up new skills automatically.
  3. Commit and push. Teammates git pull and it shows up.

How the parallel review actually works

The most important design rule, repeated here so it isn't lost: the 5 reviewer agents are spawned in a single assistant message with 5 Agent tool-use blocks. Sequential calls would defeat the entire point of the skill.

Each agent gets:

  • The same diff (or plan) inlined into the prompt — agents don't share filesystem context with the host.
  • A different lens (correctness / security / error handling / readability / testing for code; goal alignment / technical soundness / risk / testability / decomposition for plans).
  • An explicit "you are reviewer N of 5, do not collaborate, give your honest read" framing.
  • A strict return-format contract — RATING:, TOP_ISSUES: with bracketed severity tags, SUMMARY: — so the host can table-ize without parsing prose.

The host Claude then:

  • Parses each reviewer's structured return (malformed returns are noted in the table, not retried).
  • Shows the rating spread rather than averaging — disagreement is signal.
  • Surfaces unanimous concerns (≥3 reviewers, deduped semantically).
  • Calls out contested calls explicitly.
  • Synthesizes one recommendation: ship, ship-with-fixes, or do-not-ship.
